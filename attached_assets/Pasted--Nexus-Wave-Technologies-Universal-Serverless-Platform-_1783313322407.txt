## 6. No Source-Code Modification for New API Configuration

यह एक critical requirement है।

नई API key, AI model, external service endpoint या provider configuration add करने के लिए application source code modify, rebuild या redeploy करने की आवश्यकता नहीं होनी चाहिए, बशर्ते उस provider के लिए compatible generic adapter या protocol adapter पहले से supported हो।

Provider system configuration-driven बनाओ।

Configuration environment variables, deployment secrets या encrypted Admin configuration से load हो।

उदाहरण configuration model:

PROVIDER_NAME

PROVIDER_TYPE

PROVIDER_BASE_URL

PROVIDER_API_KEY

PROVIDER_AUTH_TYPE

PROVIDER_MODEL

PROVIDER_TIMEOUT

PROVIDER_ENABLED

PROVIDER_PRIORITY

PROVIDER_CAPABILITIES

PROVIDER_REQUEST_FORMAT

PROVIDER_RESPONSE_MAPPING

PROVIDER_RATE_LIMIT

PROVIDER_FALLBACK

Exact naming implementation के अनुसार बदल सकती है।

लेकिन architecture ऐसा हो कि supported protocol family की नई service जोड़ने के लिए केवल:

* Secret add करना
* Base URL add करना
* Model name add करना
* Provider type select करना
* Provider enable करना

पर्याप्त हो।

Source code modification आवश्यक न हो।

Important security rule:

Arbitrary environment variable को automatically executable API configuration मत मानो।

केवल validated provider schema, approved protocol adapter और secure configuration registry के माध्यम से provider enable करो।

## 7. Generic Provider Adapter Architecture

Backend में reusable adapters बनाओ।

Examples:

* OpenAI-compatible adapter
* Generic REST JSON adapter
* OAuth API adapter
* API-key header adapter
* Bearer-token adapter
* Query-key API adapter
* Image-generation adapter
* Text-generation adapter
* Structured-data API adapter
* Webhook adapter

यदि कोई नई AI service OpenAI-compatible protocol support करती है, तो केवल environment configuration बदलकर उसे connect किया जा सके।

यदि कोई REST API generic supported configuration से map हो सकती है, तो source modification के बिना configuration से connect हो सके।

यदि कोई provider completely unique protocol use करता है, तो उसके लिए एक बार adapter development आवश्यक हो सकता है।

उसके बाद उसी protocol family के नए providers configuration से add किए जा सकें।

यह limitation documentation में साफ लिखो।
